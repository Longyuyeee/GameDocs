# GameDocs 采集守则 v2.0

> 本守则是面向**所有品类、所有游戏**的全局强制性采集规范。各品类的具体采集策略（维度、清单、流程）为补充，与本守则冲突时以本守则为准。
>
> 制定依据：39款游戏的完整采集经验总结。

---

## 一、核心原则

### 1.1 三条铁律

1. **每一个数字必须有来源** — 没有来源 URL 的数据不允许写入文档
2. **文档深度优先于文档数量** — 一篇 5KB 的详细数据表胜过三篇 1KB 的空壳文档
3. **真实数据优先于善意推测** — 无法验证的数据标注"待补充"，不凭空捏造

### 1.2 采集的本质

采集不是"把网上能找到的信息抄下来"——是**将零散的、多语言、多来源的信息，整合为结构化、可查询、可对比的设计参考文档**。

一篇合格的采集文档 = **准确的数据表 + 清晰的来源标注 + 设计意图分析 + 跨游戏对比视角**。

---

## 二、质量等级标准

### 2.1 文档质量三级制

| 等级 | 标志 | 机制文档标准 | 数据文档标准 |
|:--:|:--:|------|------|
| **A 级** | ⭐ | ≥10,000 字节，含 5+ 数据表，含设计意图分析，含跨游戏对比 | ≥5,000 字节，全部为结构化表格，含性价比/策略分析 |
| **B 级** | — | ≥3,000 字节，含 3+ 数据表，含设计特点分析 | ≥2,000 字节，主要数据以表格呈现，含来源标注 |
| **C 级** | ⚠ | <3,000 字节或纯文字无表格 | <2,000 字节或数据稀疏 |

- **目标**：所有游戏达到 B 级以上。品类标杆（星露谷物语/梦幻魔法公主）达到 A 级。
- **红线**：任何文档不得为 C 级——宁缺毋滥。

### 2.2 文档深度检查清单

在声称"采集完成"之前，逐项检查：

- [ ] 每份文档是否标注了**游戏版本号**？
- [ ] 每份文档是否标注了**数据来源 URL**（不少于 2 个独立来源）？
- [ ] 机制文档是否包含**"设计特点分析"段落**（不只描述机制，还要分析设计意图）？
- [ ] 数值文档是否以**结构化 Markdown 表格**为主体（非纯文字段落）？
- [ ] 关键数值是否有**跨游戏对比**或**同品类参照**（至少 1 处）？
- [ ] 文档是否包含**"本游戏独有/特色"的识别**？
- [ ] 缺失的数据项是否标注了**"待补充"**而非留空？

## 三、数据来源规范

### 3.1 来源优先级

```
L1  官方 Wiki / 攻略本            ← 最权威，数据矛盾时的最终仲裁者
L2  社区 Wiki (Fandom/Wiki.gg)    ← 信息量最大，需交叉验证
L3  Steam 社区指南 / GameFAQs     ← 玩家验证过，质量参差不齐
L4  B站 / YouTube 攻略视频        ← 可视化参考，数字需截图核对
L5  贴吧 / NGA / Reddit / TapTap  ← 社区共识，需要多源交叉验证
L6  知乎 / 豆瓣 等评测分析        ← 设计分析优先，数值仅供参考
L7  数据挖掘 / 解包工具           ← 最精确但可及性最低
```

### 3.2 来源标注格式

每份文档**开头**必须包含：

```markdown
> 游戏版本: [游戏名称] v[版本号] (平台, 年份)
> 数据来源: [来源1简称](URL1), [来源2简称](URL2), ...
```

- 来源数量：机制文档 ≥2 个，数据文档 ≥3 个
- URL 必须完整可访问（不短链、不外链已失效页面）
- 版本号不可省略（即使只能推断大致版本）

### 3.3 数据矛盾处理

1. 以更高优先级的来源为准（L1 > L2 > ... > L7）
2. 在文档中标注差异：如 `售价: 150G (Wiki) / 140G (Steam指南)`
3. 如果两个同等级来源矛盾，**优先取官方 Wiki 的值**

## 四、文档内容规范

### 4.1 命名规则（全局统一）

| 类型 | 规则 | 示例 |
|------|------|------|
| 游戏文件夹 | 中文（有通用中文译名则使用） | `星露谷物语/` `火山的女儿/` |
| 机制文档 | 中文描述，不使用英文 | `日程与行动系统.md` |
| 数值文档 | 中文 + "数据总览"后缀 | `活动属性数据总览.md` |
| 策略/索引文件 | 英文下划线前缀 | `_index.md` |
| 所有文件夹 | 中文 | `机制分析/` `数值数据/` `特色文档/` |

### 4.2 禁止的行为

| 禁止 | 原因 | 正确做法 |
|------|------|------|
| 凭空编造数值 | 破坏信任，失去参考价值 | 标注"待补充"或"未公开" |
| 只复制粘贴不整理 | 无法形成结构化参考 | 按模板重新组织数据 |
| 忽略年龄/版本差异 | 不同版本数值可能不同 | 标注版本号，多版本有差异时注明 |
| 纯文字无表格 | 数据不可查询 | 提取数字→制作表格 |
| 只写"是什么"不写"为什么" | 缺少设计参考价值 | 添加设计特点分析 |
| 不标注来源 | 不可验证 | 至少标注 2-3 个来源 URL |
| 文档过短(<2KB)  | 深度不足，无参考价值 | 要么加深到 ≥3KB，要么合并到相关文档 |

### 4.2b 游戏概览页（强制要求）

**每款游戏必须在其根目录创建 `游戏概览.md`**，作为从品类索引导入的着陆页。该文件必须包含：

```markdown
# [游戏名称] — 游戏数据总览

> 所属品类: [品类链接] | 文档数: X 份 | 总大小: X KB

## 一、游戏简介
[发行年份/开发商/平台/核心玩法概述/独特机制——约100-200字]

## 二、文档统计
| 目录 | 文档数 | 大小 |
|------|:--:|:--:|
| 机制分析 | X 篇 | ~X KB |
| 数值数据 | X 篇 | ~X KB |
| 特色文档 | X 篇 | ~X KB |

## 三、机制分析文档（带大小和说明的完整列表）
## 四、数值数据文档（同上）
## 五、特色文档（同上, 含必有的 团队与开发历程.md）
```

**要求**:
- 游戏简介必须包含发行年份、开发商、平台、核心玩法概述、至少 1 个独特机制
- 每个文档链接必须可点击跳转
- 每个文档标注文件大小（KB）
- 特色文档中必须包含 `团队与开发历程.md`
- 采集完成一个游戏后**立即生成**此文件
- 此文件应使用脚本自动生成基础结构，再手动补充游戏简介和文档说明

### 4.3 鼓励的行为

| 鼓励 | 说明 |
|------|------|
| 跨游戏对比 | "与 XXX 不同，本游戏 YYY 采用了..."（至少每篇机制文档 1 处） |
| 性价比/策略分析 | 不只罗列数据，还要指出"最优选择""陷阱选项""推荐策略" |
| 标注游戏独有特色 | 每款游戏识别其独有机制深入分析，数量不限——有一篇写一篇，有十个写十个。复杂游戏（如Elona/RimWorld）的独有特色文档可能多达 5-8 篇 |
| 注明数据局限性 | "以下数据基于 v1.X / 未验证"→对读者诚实 |
| 保留英文/日文原名 | 首次出现时括号附注，便于读者去外网搜索 |

### 4.4 团队与开发文档（强制要求）

**每款游戏必须在 `特色文档/` 中创建 `团队与开发历程.md`**。

#### 文档结构

```markdown
# 团队与开发历程 — [游戏名称]

> 开发商/工作室
> 发行年份 | 平台 | 团队规模

## 1. 开发团队
- 核心成员及其角色
- 团队规模（3人/10人/50人+等）
- 是否有外包或外部合作

## 2. 开发历程
- 开发周期（从立项到发售）
- 关键里程碑
- 遇到的重大挑战及如何解决
- 是否经历过延期、重做、资金困难等

## 3. 技术与工具
- 使用的引擎和关键工具
- 美术/音频制作流程（如有信息）
- 团队协作方式（如有信息）

## 4. 市场表现
- 销量数据（如有）
- Steam评价数据（好评率/评价数）
- 获奖记录

## 5. 设计启示
- 对独立游戏开发者的启示
- 对养成/牧场经营品类的参考价值
- 团队规模与游戏品质的关系分析
```

#### 重点关注的游戏类型

| 优先级 | 游戏类型 | 为什么重要 |
|:--:|------|------|
| ★★★★★ | **小团队成功案例**（3-10人） | 最具参考价值——证明了小团队在养成/牧场品类中可以做出爆款 |
| ★★★★ | **独立游戏里程碑** | 单人开发或极小团队的品类突破 |
| ★★★ | **中等团队**（10-50人） | 工业化流程与独立精神的平衡 |
| ★★ | **大厂/大型团队**（50人+） | 作为对比基准 |

#### 信息来源优先级

```
1. 开发者访谈/GDC演讲/Postmortem  ← 最权威（一手信息）
2. 官方新闻稿/开发者博客/Twitter   ← 直接信息
3. 游戏媒体专访/评测中的开发背景   ← 二手但可靠
4. Steam商店页/官方网站            ← 基础信息
5. 社区推测（标注"未经证实"）       ← 最低优先级
```

> 如果某款游戏的开发信息极少（如年代久远的游戏），文档中应诚实标注"信息有限"并仅填写可验证的部分，不凭空编造。

## 五、品类差异与适应

### 5.1 文档维度由游戏设计驱动（v2.1 更新）

**核心原则：品类策略文档中的维度列表是采集起点和最低覆盖要求，不是文档数量上限。**

| 游戏类型 | 文档调整原则 |
|------|------|
| **标准游戏** | 品类策略文档中的维度列表为**最低覆盖要求**。复杂游戏可以且应该超出（如将一个大维度拆为 2-3 篇独立文档），由游戏实际的设计维度决定最终文档数量 |
| **跨界游戏** | 保持核心维度覆盖，但维度内容可针对性调整；数据表可适当减少 |
| **纯叙事/边界游戏** | 文档维度完全重新适配，不强制套用标准模板 |

**判断文档数量的唯一标准**：这款游戏有多少个**值得独立分析的设计维度**？如果一个系统足够复杂、独特、有参考价值，就值得独立成篇——无论它是品类策略中的第 11 个还是第 15 个维度。

### 5.2 文档数量不是指标（深化）

- 有 10 篇每篇 3KB 的文档 ≠ 完成了高质量采集
- 有 5 篇每篇 8KB 的文档 = 更参考价值的输出
- **永远以文档深度（字节数+数据密度）而非文档数量来评价质量**
- **同理，不应该因为"模板只有 N 篇"就限制复杂游戏的文档数量**——Elona 值得 12 篇机制分析，就写 12 篇
- 数值数据文档数量**跟随机制分析文档数量**——每个独立机制维度应有对应的数据总览

## 六、采集流程

### 6.1 新游戏采集标准流程

```
阶段 1: 信息搜集（至少使用 3 个独立来源）
  1. 搜索官方 Wiki（L1）——获取权威数值基础
  2. 搜索社区 Wiki + Steam 指南（L2-L3）——补充细节和验证
  3. 搜索中文攻略（贴吧/B站/NGA）（L5）——获取中文社区共识
  4. 搜索评测/分析文章（L6）——获取设计解读

阶段 2: 文档编写
  5. 先写机制分析（理解系统→再写数据表效率更高）
  6. 每篇机制文档至少包含 3 张数据表 + 设计特点分析
  7. 再写数值数据（从机制文档中提取数字→制作结构化表格）
  8. 最后写特色文档（识别游戏的独有机制深入分析，数量不限，**其中必须包含 `团队与开发历程.md`**）
  9. **生成游戏概览页**（`游戏概览.md`）——包含游戏简介、文档统计、每个文档的大小和可点击链接

阶段 3: 质量检查
  10. 逐项核对"文档深度检查清单"
  11. 检查字节数：机制≥3KB / 数据≥2KB（B 级最低标准）
  12. 检查链接：来源 URL 可访问 / 内部链接不失效（含游戏概览页中的所有链接）
  13. 跨游戏对比：至少 1 处与同品类其他游戏的对比
  14. **更新品类 _index.md 中的链接指向游戏概览页**
```

### 6.2 已有游戏补充流程

```
1. 先检查文档深度（字节数 + 表格密度）
2. 识别最薄弱的文档→优先加深
3. 搜索最新数据（游戏可能已更新）
4. 补充缺失的来源 URL
5. 检查机制文档是否含"设计特点分析"
```

## 七、文档维护

### 7.1 版本更新

- 游戏推出重大更新（如 DLC、重制版）→ 标注新旧版本差异
- 不删除旧版本数据，添加 `v1.X (旧)` vs `v2.X (新)` 对比

### 7.2 来源链接失效

- 定期检查外部链接可访问性
- 失效链接标注 `[已失效]` 并尝试寻找替代来源

### 7.3 质量标准升级

- 本守则版本号见文末
- 随着采集实践的深入，标准持续提高
- 每次标准升级需更新本文件并标注改动

## 八、经验总结：常见错误与教训

### 8.1 常见错误

| 错误 | 表现 | 预防 |
|------|------|------|
| **空壳文档** | 机制文档有 10 篇但每篇仅 1-2KB——看起来覆盖了所有维度，实际上每篇都没内容 | 宁缺毋滥：宁可 5 篇厚文档也不要 10 篇薄文档 |
| **无源之水** | 文档有大量数值但没有来源标注——不可验证 | 强制要求每份文档开头标注来源 URL |
| **复制粘贴** | 从 Wiki 直接复制文字，不做整理消化——不是"采集"而是"搬运" | 用自己的结构重新组织数据，添加分析和对比 |
| **忽略更新** | 文档标注的游戏版本过旧——数据可能已不准确 | 定期检查游戏更新（尤其是 EA 游戏） |
| **翻译丢失** | 只使用英文来源，忽略了中文社区的大量攻略和讨论 | 中英文来源平衡使用 |
| **只看数量** | 关注"做了多少文档"而非"每篇文档有多少内容" | 以字节数和表格密度为质量指标 |

### 8.2 关键教训

1. **文档深度比文档数量重要 10 倍** — 这是本项目的核心教训。经历过 39 款游戏的采集后，最"薄"的文档大多是最早采集的——当时只关注"覆盖所有维度"，忽视了每篇文档的内容深度。
2. **没有来源的数据 = 没有价值** — 如果未来的读者（或一年后的自己）无法判断数据的准确性，整份文档的参考价值归零。
3. **设计分析 = 采集的灵魂** — 单纯罗列数据是 Wiki 的工作。GameDocs 的价值在于"解读"——分析设计意图、对比同类机制、提取可复用的设计模式。
4. **跨界游戏的维度需要灵活调整** — 强行套用标准模板（如对纯 VN 游戏要求"日程与行动系统"）只会产生低质量的填充文档。识别游戏的本质类型，据此调整文档维度。
5. **信息可及性不同 → 采集深度也应不同** — 有官方 Wiki 的游戏（星露谷/符文工房）可以做到深度数据表，缺乏公开资源的游戏（莱尔兰德物语）应该诚实标注局限性。
