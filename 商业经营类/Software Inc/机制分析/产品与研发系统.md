## 二、完整开发生命周期(4阶段)

| 阶段 | 核心活动 | 时长(典型值) | 关键角色 | 产出物 |
|:----:|---------|:----------:|---------|-------|
| **1. Design(设计)** | Lead Designer创造产品概念→选择功能特性→决定目标平台→设定技术规格 | 数周 | Lead Designer(Creativity属性决定创意质量上限) | 设计文档(Design Document) |
| **2. Alpha(开发)** | 程序员+艺术家实现功能→Design迭代→Peer Review(同事评审)→Bug产生 | 数月-数年(视规模) | 程序员/艺术家/全体开发团队 | 可运行Alpha版本 |
| **3. Beta(测试)** | Bug修复→打磨优化→性能调优→可随时发布 | 数周-数月 | 全员 | Beta版→正式版 |
| **4. Release→Support(支持)** | 发布→持续更新→修复Bug→维持口碑→可推出补丁/DLC | 持续(数月-数年) | 部分开发+支持团队 | 补丁/更新/DLC |

### 各阶段关键机制

#### Design阶段

| 机制 | 说明 |
|------|------|
| **Lead Designer指定** | 项目启动前必须指定Lead Designer。Creativity属性决定产品"创意质量"上限 |
| **功能特性选择** | 不同类型软件有不同功能需求——OS需要"文件系统""GUI""网络协议"；游戏引擎需要"物理引擎""渲染管线"等。选错功能=浪费时间+产品竞争力不足 |
| **目标平台决策** | 决定产品发布在哪个OS上(抢占平台生态位) |
| **技术规格设定** | 影响开发时间与品质上限的平衡 |

#### Alpha阶段

| 机制 | 说明 |
|------|------|
| **并行开发** | 多个任务可同时进行，取决于团队规模和分工 |
| **Peer Review(同事评审)** | Alpha阶段中定期触发的评审事件——同事检查代码/设计→发现Bug→修复。评审质量取决评审者技能 |
| **Bug产生** | 开发过程中自然产生Bug。数量=功能复杂度×团队平均技能(技能低→Bug多) |
| **Task分配** | 玩家可将具体任务分配给特定员工——粒度精细的管理选项 |
| **Design迭代** | 开发过程中Design阶段可调整——但调整越晚成本越高 |

#### Beta阶段

| 机制 | 说明 |
|------|------|
| **Bug修复** | 修复所有已知Bug → 质量提升 |
| **打磨(Polish)** | 额外投入时间提升品质——边际效益递减 |
| **随时发布** | Beta阶段可随时选择发布——越早发布→Bug越多/评分越低/口碑越差；越晚发布→品质越高/但时间成本越高 |
| **"Ship or Polish"决策** | 核心权衡决策——再打磨一个月可能品质+2%，但竞争对手可能抢先发布 |

## 三、Lead Designer系统

Lead Designer是Software Inc.最独特的品类级设计：

| 属性 | 影响 |
|:---:|------|
| **Creativity(创造力)** | 决定产品创意质量上限。高Creativity→设计文档质量高→产品评分上限高 |
| **专业化匹配** | Lead Designer的专业化应与产品类型匹配——System设计师领导OS开发，2D设计师领导图像编辑器开发 |

> Lead Designer不能是随便谁——这是项目启动的"硬门槛"。如果公司没有合适的Lead Designer(没有招聘或技能不匹配)，就无法启动该类型产品的开发项目。

## 四、技术等级与专利系统

### 技术研究

| 机制 | 说明 |
|------|------|
| **Tech Level(技术等级)** | 每个技术领域有独立的等级。通过安排员工研究提升 |
| **研究时间** | 研究新科技需要投入员工时间(折合工资成本) |
| **技术树** | 不同技术领域有前置依赖关系——解锁低级科技后才能研究高级科技 |

### 专利系统

| 机制 | 说明 |
|------|------|
| **研究→专利** | 技术研究完成→自动获得专利(或需手动申请，待确认) |
| **专利授权版税** | 将专利授权给其他公司使用→获得持续版税收入(被动收入核心来源之一) |
| **购买他人专利** | 可使用其他公司的专利(需支付授权费)——降低研发投入但增加运营成本 |
| **技术壁垒** | 持有核心专利→形成竞争壁垒→阻止AI公司使用该技术 |
| **交叉授权** | 待补充(可能支持公司间交叉授权协议) |

## 五、产品生命周期管理

| 机制 | 说明 |
|------|------|
| **续作(Sequels/Follow-up)** | 基于前作共享特性→缩短开发时间+维持品牌热度 |
| **移植(Porting)** | 发布后可将产品移植到其他OS平台→扩大市场覆盖 |
| **DLC/Add-ons** | 为已发布产品添加新内容→延长生命周期+增加收入 |
| **Franchise/系列化** | 长期经营的产品线品牌积累效应——系列续作天然享有粉丝基础 |
| **版本更新** | 发布后持续更新(修Bug+小功能)→维持口碑和评分 |
| **折扣促销** | 发布后可临时降价→短期销量拉升 |
| **产品停售(EOL)** | 旧产品最终会被市场淘汰→自动下架或主动终止支持 |

## 六、品质与评分体系

| 评分维度 | 说明 | 提升手段 |
|---------|------|---------|
| **设计质量** | 产品功能/创意/用户体验 | Lead Designer的高Creativity+好的功能选择 |
| **技术质量** | 代码质量/稳定性/性能 | 高技能程序员+充分的Beta打磨 |
| **视觉/音频质量** | UI设计/图标/音效 | 高水平艺术家/音频设计师 |
| **Bug率** | 负面指标——Bug越多品质越低 | Peer Review+充分测试+修复 |
| **创新度** | 待补充 | 待补充 |

> 综合评分 = 各维度加权汇总。不同软件类型各维度权重不同——OS更重设计+技术，游戏更重视觉+创新。

## 七、设计特点分析

**"Ship or Polish"的核心困境**：Beta阶段"随时可发布但品质不一定达标"的设计，创造了一个贯穿游戏始终的策略困境——早发布=抢市场但低评分损害口碑；晚发布=高品质但可能错过窗口期+消耗更多研发成本。这不是对错题，而是风险权衡。这种设计模拟了真实软件行业的"发布决策压力"——而大多数经营游戏完全省略了这个困境。

**Lead Designer的品类壁垒设计**：要求在项目启动前就有合适的Lead Designer——这不是个简单的领导角色，而是一条"品类壁垒"：你不能凭空进入一个你不熟悉的软件开发领域。这个机制迫使玩家逐步积累团队能力，而非一开始就什么都做。与Game Dev Tycoon的"点主题→直接开做"相比，Software Inc. 的Lead Designer系统增加了策略深度的同时，也提高了入门的认知门槛。

**Design-Develop-Test的3阶段真实感**：绝大多数游戏开发模拟器将开发简化为"一条进度条"。Software Inc. 将其拆分为Design(概念规划)→Alpha(功能实现)→Beta(质量打磨)三个阶段——每个阶段有独立的活动、角色和决策点。这种"阶段分离"设计让玩家理解软件开发不仅仅是"写代码"，设计规划和测试打磨同等重要。

## 八、跨品类对比

**vs Game Dev Tycoon**: GD Tycoon的开发流程是"概念→制作→打磨"三段推进条——每个阶段自动进行，玩家只能分配员工类型(设计/技术/平衡)。Software Inc. 的开发流程细致到功能特性选择、Lead Designer指定、Peer Review安排、Bug追踪修复等——复杂度是GD Tycoon的10倍以上。这种差异反映了两款游戏的目标用户群：GD Tycoon是"简化的游戏开发体验"，Software Inc. 是"真实的软件企业管理模拟"。

**vs 开罗游戏开发物语**: 开罗系列的开发是"开发→评分→奖项→续作"的更简化循环——连开发阶段都没有细分，本质是一个"投钱→出分"的抽奖机制。SoftWare Inc. 在这个维度与开罗系列处于完全不同的复杂度层级。

**vs 双点医院**: 双点医院的"研发"主要是研究新设备/新治疗手段，没有产品生命周期概念。两者唯一的共同点是都需要前期研究投入——但Software Inc. 的研发产出是一个独立产品(可销售)，双点医院的研发产出是"可用的设备"(服务于运营效率提升)。
