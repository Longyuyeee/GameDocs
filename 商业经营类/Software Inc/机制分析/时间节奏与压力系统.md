# 时间节奏与压力系统 — Software Inc.

> 游戏版本: Software Inc. Beta 1.8 (PC/Steam, 2015+, Coredumping)
> 数据来源: [Software Inc. Wiki (Fandom)](https://software-inc.fandom.com), [官方博客](https://softwareinc.coredumping.com), [Steam社区指南](https://steamcommunity.com/app/362620/guides/)

## 一、时间结构

| 时间单位 | 对应现实 | 游戏内事件 |
|---------|---------|-----------|
| **1 小时** | 实时(可加速) | 员工工作/休息循环 |
| **1 天** | ~8小时(工作时间) | 员工通勤→上班→工作→休息→下班 |
| **1 月** | 约30天 | **工资发放日(最大支出)**、账单结算 |
| **1 年** | 12个月 | 技术世代更替、市场趋势变化、员工年龄增长 |
| **游戏全程** | 1980年起→无限 | 开放式沙盒，无时间上限 |

### 员工日程表

| 时间段 | 活动 | 影响 |
|:-----:|------|:----:|
| 上午 8:00-9:00 | 通勤/到岗 | 通勤时间影响效率 |
| 上午 9:00-12:00 | 工作时段 | 核心开发时间 |
| 中午 12:00-13:00 | 午餐休息 | 食堂需求→食堂距离影响休息效率 |
| 下午 13:00-17:00 | 工作时段 | 核心开发时间 |
| 17:00+ | 下班(标准班) | 加班需额外支付/影响满意度 |
| 夜班(可选) | 22:00-6:00 | 需设置夜班制度，效率较低 |

> **注意**: 员工日程不是固定的——如果食堂太远，员工可能花1小时在路上、只吃10分钟就回去工作。物理空间布局直接影响时间利用效率，这是 Software Inc. 独有的设计耦合。

## 二、开发周期节奏

| 阶段 | 时长(估) | 核心工作 | 资源消耗 | 输出 |
|:---:|:--------:|---------|:-------:|-----|
| **Design** | 几天-2周 | 定义产品规格、功能集 | Lead Designer时间 | 产品蓝图(Feature List) |
| **Alpha** | 几周-几个月 | 程序员编码+艺术家制作资源 | 全团队全力投入 | 可运行但未完成的产品 |
| **Beta** | 1-4周 | Bug修复、性能优化、打磨 | 全团队(侧重QA) | 接近发布版本 |
| **Release** | 1天 | 正式发布 | 营销+发布操作 | 上市产品 |
| **Support** | 无限(可选) | 持续更新、Bug修复、DLC | 缩减后团队 | 补丁/更新/资料片 |

### 各阶段资源消耗权重

| 阶段 | 工资消耗 | 研发费用消耗 | 营销费用消耗 | 现金流状态 |
|:---:|:-------:|:-----------:|:-----------:|:---------:|
| Design | 低 | 低 | 无 | 正常 |
| Alpha | 高 | 高 | 低(可开始预热) | **最紧张** |
| Beta | 高 | 中 | 中(开始营销) | 紧张 |
| Release | 中 | 低 | **最高**(冲刺营销) | 即将回血 |
| Support | 低 | 低 | 低 | 正现金流 |

## 三、市场压力——时间的无形对手

### 技术世代演进

| 时代 | 大致年份 | 热门品类 | 技术特征 | 玩家策略 |
|:---:|:-------:|---------|---------|---------|
| 早期 | 1980s | 基础工具、简单游戏 | 2D图形、命令行界面 | 积累初始资本 |
| 增长期 | 1990s | OS爆发、办公软件、3D游戏兴起 | 图形界面、3D渲染 | 进入主流市场 |
| 互联网期 | 2000s | 网络工具、杀毒软件、游戏 | 互联网功能、在线服务 | 拓展产品线 |
| 移动期 | 2010s | 移动应用、手游 | 触屏、便携设备 | 进入移动市场 |
| 未来 | 2020s+ | 取决于玩家发展方向 | AI/VR/新硬件 | 定义市场方向 |

### 先发优势与市场窗口

| 场景 | 效果 | 应对策略 |
|------|------|---------|
| 玩家率先发布新产品 | **先发优势**、高市场份额 | 抢占未开发品类 |
| AI公司抢先发布同类产品 | 份额被瓜分、利润降低 | 差异化或价格战 |
| 市场趋势变化 | 旧品类需求衰退 | 提前布局新兴品类 |
| 技术过时 | 旧产品销量持续下滑 | 发布新版本/重制版 |

## 四、现金流压力——最持续的威胁

### 月度固定支出结构

| 支出项 | 占比(估) | 弹性 | 说明 |
|-------|:-------:|:----:|------|
| **工资** | **60-80%** | 低 | 每月的最大固定支出 |
| 办公室租金 | 5-15% | 低 | 取决于办公室规模 |
| 设备维护 | 3-8% | 中 | 电脑/服务器/空调等 |
| 电费/水费 | 2-5% | 中 | 取决于办公室面积和设备 |
| 贷款利息 | 0-20% | 高(取决于贷款) | 如果有贷款则为重大负担 |
| 营销费用 | 可变 | 高 | 可控制投入 |

### 现金流危机阶段

| 阶段 | 现金储备 | 风险等级 | 应对方案 |
|:---:|:-------:|:-------:|---------|
| 安全 | ≥6个月工资 | 低 | 正常运营 |
| 警告 | 3-6个月工资 | 中 | 考虑接合同 |
| **危险** | 1-3个月工资 | **高** | 必须接合同或裁员 |
| **致命** | <1个月工资 | **极高** | 银行贷款或出售IP |

### 贷款系统

| 贷款类型 | 额度 | 利率 | 期限 | 风险 |
|---------|:---:|:---:|:---:|:----|
| **快贷(Quick Loan)** | ~$1M | 高利率 | 短期 | 快速过桥但成本高 |
| **银行贷款(Bank Loan)** | ~$10M | 较低利率 | 长期 | 额度大但审核严 |
| **个人借贷** | 可变 | 可变 | 可变 | 取决于声誉 |

## 五、年龄压力——员工的时间漏斗

| 员工年龄 | 阶段 | 效率 | 风险 |
|:-------:|:---:|:---:|:----|
| 20-35 | 职业上升期 | 高 | 低(但可能在积累经验后跳槽) |
| 35-50 | 职业巅峰 | 最高(经验加成) | 中(工资高) |
| 50-65 | 经验丰富但效率下降 | 中 | 高(即将退休) |
| 65+ | **退休** | 退休 | 知识流失 |

### 知识传承策略

| 策略 | 操作 | 效果 | 成本 |
|-----|------|------|:---:|
| 老带新 | 老员工与新员工协同工作 | 新员工学习速度加快 | 占用老员工时间 |
| 文档记录 | 将老员工知识文档化 | 离职后知识保留 | 耗费时间编写文档 |
| 培训课程 | 正式培训体系 | 系统化知识传递 | 培训费用 |
| 招聘替代 | 直接招聘有经验者 | 快速补充 | 高工资成本 |

> **设计意图**: 员工会老去和退休——这意味着你无法"一次招聘，用一辈子"。这迫使玩家持续进行人才储备和梯队建设，模拟了真实企业中"关键人物风险"的管理挑战。

## 六、开放式沙盒——无"失败条件"

Software Inc. 没有传统意义上的 Game Over：

| 状态 | 发生条件 | 后果 | 能否恢复 |
|:---:|---------|------|:-------:|
| 破产 | 负债超过资产+无法偿还 | 游戏继续但无法运营 | 可能(被收购或重启) |
| 创始人退休 | 年龄过大 | 可以指定继承人 | 可继续 |
| 被收购 | 持股被买断 | 成为子公司 | 可继续(但控制权变化) |
| 员工全部离职 | 极度低满意度 | 公司瘫痪 | 可重新招聘 |

## 八、设计特点分析

**时间作为核心资源维度**: 在许多商业经营游戏中（如开罗系列），时间是"等待进度条走完"的被动维度。在 Software Inc. 中，时间是主动管理的资源——员工从工位到食堂的距离、从家到公司的通勤时间、加班与满意度的权衡，所有这些都让时间从抽象指标变成了可优化的物理变量。

**三层压力系统的叠加效应**: Software Inc. 有三层独立的压力系统——市场竞争(外部)、现金流(财务)、员工老化(人力)——它们互相叠加而非独立运作。当市场进入衰退期时，可能正好遇到现金流紧张，而核心员工又面临退休。这种压力叠加效应让游戏的中后期保持紧张感，而非"一旦做大就无敌"。

**开放式设计的哲学**: 没有 Game Over、没有强制终局、玩家可以无限玩下去——这体现了 Software Inc. 的"沙盒优先"设计哲学。压力系统不是为了"让玩家失败"，而是为了"创造有意义的决策情境"。即使破产了，玩家也可以选择继续运营或重新开始——失败本身就是故事的一部分。

**与 Game Dev Tycoon 的对比**: GD Tycoon 的时间系统以"完成游戏→发布→评分→赚取经验→下一款"的循环为主，时间跨度较紧凑（约30-50年）。Software Inc. 的时间跨度更长（1980年起无限），时代演进更缓慢，但每种决策的影响也更持久——建一栋楼可能需要游戏内数月，而一款产品的开发周期可长达数年。GD Tycoon 的节奏像"回合制"，每一款游戏是一个回合；Software Inc. 的节奏更像"实时战略"，多款产品并行、多团队同时运作、长期规划与短期应急并存。

## 九、与同类游戏对比

| 维度 | Software Inc. | Game Dev Tycoon | 开罗·游戏开发物语 |
|------|:------------:|:--------------:|:----------------:|
| 时间跨度 | 1980起无限 | ~30-50年(终局) | ~20-30年(终局) |
| 开发周期 | 几周-几年 | 几个月(固定) | 几个月(固定) |
| 现金流压力 | **核心机制**(工资/租金/设备) | 有(但较轻) | 有(但较轻) |
| 员工退休 | **有**(关键风险) | 无(员工不退休) | 无(员工不退休) |
| Game Over | 无(沙盒) | 无(沙盒) | 有(年限到) |
| 加班系统 | 有(影响满意度) | 无 | 无 |
| 多项目并行 | 有(多团队) | 无(单项目) | 无(单项目) |
| 时代更替 | 手动感受+市场趋势 | 平台世代更替 | 年代更替 |
