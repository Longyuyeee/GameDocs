### 1.3 茄子的"非典型开发者"背景

茄子的背景是中国独立游戏圈中最富传奇色彩的故事之一:

| 阶段 | 内容 |
|:----:|:----:|
| **教育背景** | 中文系毕业 (非计算机/游戏设计专业) |
| **原职业** | 建筑设计师 (用CAD/天正画图) |
| **第一次尝试** | 用 RPG Maker 做了一款西式奇幻题材游戏《末日文书》 |
| **转机** | 2015 年组建螺舟工作室，启动太吾绘卷项目 |
| **最大危机** | 开发一年后，团队唯一的主程序员**退出**，茄子被迫自学编程接手全部代码 |

"唯一的程序员退出"是螺舟发展史上最关键的事件。茄子当时面临的选择是: 放弃项目，或者完全不懂编程的自己接手全部代码。他选择了后者——从零开始在 Unity 中学 C#，用"完全业余的代码水平"继续开发。这批代码在后续被称为 **"绿皮代码"**。

## 2. 开发历程

### 2.1 时间线

| 阶段 | 时间 | 关键事件 |
|:----:|:----:|---------|
| 螺舟工作室成立 | 2015年 | 茄子组建远程小团队 |
| 项目启动 | ~2016年 | 太吾绘卷正式立项 |
| 主程序退出 | ~2017年 | 面临项目终止危机 → 茄子自学编程接手 |
| 远程开发期 | 2015-2018年 | 团队成员分布全国各地，早期素未谋面 |
| **Steam EA 上线** | **2018年9月21日** | 首日 3 万份 / 首周 30 万份 / 3 个月 100 万份 |
| EA 阶段 | 2018-2022年 | 持续更新和内容添加 |
| "推倒重来"决策 | 2019年9月 | 茄子决定封闭开发重新制作正式版 |
| 昆明组建团队 | 2019-2020年 | 从远程转为昆明本地 18 人团队 |
| 疫情+版号危机 | 2020-2022年 | 疫情、版号停发 263 天、主程序再次离开 |
| **正式版 1.0 发售** | **2022年9月21日** | 首日同时在线 6.4 万 → BUG 多 → 口碑暴跌 |
| 持续修复 | 2022-2024年 | "实装"计划—持续更新、修复、优化 |

### 2.2 EA 阶段的"奇迹发布"

2018 年 9 月 21 日，太吾绘卷在 Steam Early Access 上线的表现是国产独立游戏史上最具冲击力的首发之一:

| 时间 | 销量 | 里程碑 |
|:----:|:----:|:-------|
| 首日 | 3 万份 | 超出所有预期 |
| 首周 | 30 万份 | 国产独立游戏首周记录 |
| 1 个月 | 80 万份 | 攀升至 Steam 全球热销榜 |
| 3 个月 | **100 万份** | 首个国产独立游戏百万销量里程碑 |

太吾绘卷 EA 版的成功不是因为画面（Unity 初级水平）、不是因为流畅度（Bug 很多）、不是因为易上手（引导极差）——而是因为 **"设计深度的降维打击"**。茄子用一套极其复杂的系统（14 个技艺 × 数百种功法 × 随机 NPC × 世界演化）创造了一个让玩家"探索不完"的武侠世界。

### 2.3 "推倒重来"的决定 (2019)

2019 年 9 月，在 EA 版销量达到 200 万份后，茄子做出了一个令整个玩家社区震惊的决定: 推倒 EA 版，用正式团队重新开发正式版。

| 旧策略 | 新策略 |
|:-------|:-------|
| 远程 3-5 人兼职开发 | 昆明 18 人全职团队 |
| "绿皮代码"继续新增功能 | 架构重写、代码重构 |
| 快速迭代、高频更新 | 封闭开发、低频沟通 |
| 社区反馈驱动 | 完整设计蓝图驱动 |

这个决定的好处: 正式版可以有更好的代码质量和更完整的系统设计。
代价: 封闭开发 3 年，社区在 2019-2022 年间几乎感受不到游戏在"前进"。

### 2.4 正式版的阵痛 (2022)

2022 年 9 月 21 日，太吾绘卷 1.0 正式版发布。开售前几小时的期待值极高——然后遭遇了口碑暴跌:

| 问题 | 影响 |
|:----|:----|
| **大量 BUG** | 正式版发布初期存在大量影响游戏进程的 BUG |
| **UI/UX 退化** | 部分玩家认为正式版 UI 反而比 EA 版更难用 |
| **功能缺失** | 一些 EA 版已有的功能在正式版中暂时移除 |
| **引导不足** | 新手体验在内容增加后反而更不友好 |

Steam 评价从 EA 阶段的"多半好评"（80%+）一度跌至 "褒贬不一"（50%+）。螺舟工作室此后进入了漫长的修复阶段——持续一年以上的"实装"计划，逐步恢复和优化被批评的系统。

## 3. 技术与"绿皮代码"的传奇

### 3.1 什么是"绿皮代码"？

茄子在直播和采访中自称他的代码为"绿皮代码"：
- 编程水平：完全业余，用 Unity 教程学到的基础 C# 语法
- 代码结构：没有设计模式、没有架构规划、全局变量满天飞
- 命名规范：随性——变量和函数名来自即时灵感
- 可维护性：茄子自己也不一定读得懂三个月前写的代码

### 3.2 "绿皮代码"创造了什么？

尽管专业程序员看到代码会感到窒息，但茄子的代码实现了一个极其复杂的系统:

| 系统 | 复杂度 |
|:----|:------|
| 14 项技艺 × 数十种功法 | 数百种可学习的技能 |
| 随机 NPC 生成 | 每个 NPC 有身份、关系、经历、立场 |
| 世界演化 | NPC 会自主行动、结婚、生子、死亡 |
| 战斗系统 | 实时回合制 + 站位 + 破绽 + 功法相克 |

"绿皮代码"的启示：**设计的价值 > 工程的价值**。当一个人的设计智慧足够深，即使他的工程水平不足以优雅地实现，仍然可以创造出令人惊叹的游戏。

## 5. 设计启示

1. **"绿皮代码"也能产出杰作** — 茄子的代码让专业程序员头疼，但他的设计深度让玩家惊叹。这是"设计大于工程"的最有力证明

2. **EA = 双刃剑** — 2018 年的爆红给了团队资源，但 4 年封闭开发让社区与开发者几乎断了联系。EA 的"持续沟通"承诺与现实之间的落差造成了口碑损耗

3. **正式版 ≠ 终局** — 太吾绘卷的故事不是"正式版发布就结束了"的故事。正式版发布只是新阶段的开始——一个与社区重建信任的漫长过程

4. **一个人的设计愿景可以承载一个品类** — 茄子一人负责策划/程序/文案/数值的几乎全部核心设计。太吾绘卷证明了"单核驱动"在游戏开发中的有效性——但也暴露了"单核驱动"的脆弱性（主程退出导致 3 年延期）

5. **"推倒重来"需要极大的勇气** — 在 200 万销量、玩家期待值最高的时候选择重构，这不是一个理性的商业决策，但这是追求"完美作品"的设计师的选择
