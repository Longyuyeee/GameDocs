## 二、Elona 开发历程

### 时间线

| 时间 | 事件 |
|------|------|
| 2006年 | noa开始开发Elona |
| **2007年8月31日** | **Elona v1.0 正式发布**(免费) |
| 2009年10月 | 发布Flash小游戏**Elona Shooter**(Kongregate平台)用于推广 |
| 2010年11月 | **noa宣布停止开发**——因无社交媒体、几乎零反馈/零同人创作，"把时间投入一个免费游戏却一无所获"的挫败感 |
| 2010年 | noa限时公开**源代码**，允许粉丝继续开发 |
| 2012年 | **Ano犬发布Elona+**——最成功的社区分支，至今仍在更新 |
| 2016年 | noa宣布精神续作**Elin**，作为Elona十周年纪念 |
| 2024年11月 | **Elin Steam EA上线**，一炮走红 |

### 停止开发的原因(noa自述)

> "那时候还没有社交媒体，几乎没有人给我反馈或画同人图。我觉得自己只是把大量时间投入到一个免费游戏里，却什么也得不到。"

这是一个一个人的开发者在没有社区支持时的孤独困境——今天被广泛视为"Roguelike沙盒"定义性作品的Elona，它的创造者当年因缺少反馈而放弃了它。

## 三、技术与工具

| 方面 | 信息 |
|------|------|
| **编程语言** | **Hot Soup Processor (HSP)** — 极其小众的日本本地化脚本语言 |
| **游戏类型** | Roguelike RPG |
| **图形** | 2D像素风(32×32精灵图) |
| **平台** | Windows PC |
| **发布方式** | 免费下载(日语原版+英文翻译版) |

> HSP是Elona最不寻常的特征之一——一个用近乎业余的语言开发的游戏，却拥有AAA级的设计深度。这本身就证明了游戏设计的核心价值在于**系统设计**而非技术栈。

### Elona(原版)

| 指标 | 说明 |

| 影响力 | 被广泛认为是太吾绘卷/Elin等后继者的灵感来源 |

### Elona的遗产

1. **定义了"Roguelike沙盒"子类型** — "你可以做任何事"的极端自由设计
2. **证明了1人开发的免费游戏可以达到商业作品的设计深度**
3. **影响了中国独立游戏** — 太吾绘卷的开发者茄子公开表示受到Elona启发
4. **HSP语言的传奇** — 用最"业余"的工具做出了最具深度的作品

## 七、设计启示

### 对独立开发者的启示

1. **一个人可以创造传奇** — noa在2007年用HSP语言一个人做出了至今仍有活跃社区的游戏。单体开发者+小众工具≠低质量作品。
2. **社交媒体改变了独立游戏** — 如果2007年有Twitter/Steam社区，noa可能不会放弃Elona。Elin的成功证明了"同样的设计智慧+现代发布渠道=完全不同的结果"。
3. **开放源代码=社区的永生** — noa的源代码公开让Elona+得以诞生，让这个游戏多活了15年以上。这是一种另一种形式的"传承"。
4. **17年的坚持终有回报** — 从2007的孤独开发者到2024的Steam热销，noa的故事是"长期主义"的终极案例。

### 对人生沙盒品类的参考价值

- Elona是一个"设计先于技术"的案例——HSP语言的限制没有阻止它成为最有深度的Roguelike沙盒。关键不在于用了什么引擎，在于系统如何交互。
